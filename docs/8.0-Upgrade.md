## What's New

### Smarter concurrency for mixed CPU and I/O workloads

Puma 8 adds opt-in support for I/O-bound request offloading with `max_io_threads` and `env["puma.mark_as_io_bound"]` ([#3816](https://github.com/puma/puma/pull/3816), [#3777](https://github.com/puma/puma/issues/3777)). This is useful for requests that do a bit of Ruby work and then spend most of their lifetime waiting on an upstream service, streaming, or other external I/O. Instead of letting those requests consume your normal CPU-oriented thread budget, Puma can temporarily grow extra threads for them.

Configure the extra headroom in `puma.rb`:

```ruby
threads 5, 5
max_io_threads 5
```

Then, once a request becomes mostly I/O wait, mark it from your Rack app:

```ruby
env["puma.mark_as_io_bound"].call
```

That combination gives you up to 5 regular request threads plus up to 5 extra I/O-bound threads. If your app has a mix of fast CPU work and slow external waits, this can improve throughput without forcing you to raise `max_threads` for everything.

Puma 8 also adds `Puma::Server#update_thread_pool_min_max` for changing the thread pool size at runtime ([#3658](https://github.com/puma/puma/pull/3658)). This is especially handy if you embed Puma, write plugins, or build control-plane tooling around it. A running server can now do `server.update_thread_pool_min_max(min: 2, max: 8)` instead of requiring a restart just to retune thread counts.

### Cleaner configuration for single and cluster mode

The new `single do ... end` and `cluster do ... end` DSL hooks make it much easier to keep one config file that behaves well in both local/single-process setups and clustered production deployments ([#3621](https://github.com/puma/puma/pull/3621)). These hooks run after config files are loaded and after Puma knows the final worker count, including values coming from `WEB_CONCURRENCY`.

That means you can keep mode-specific config together instead of splitting files or writing ad-hoc conditionals:

```ruby
workers Integer(ENV.fetch("WEB_CONCURRENCY", 0))

single do
  silence_fork_callback_warning
end

cluster do
  prune_bundler
end
```

This is a small feature, but a nice quality-of-life improvement if you share one `config/puma.rb` across development, staging, and production. Use these hooks to change mode-specific settings; keep `workers` itself outside the hooks.

### Better operations and debugging

If you use `shutdown_debug`, Puma 8 makes it easier to leave enabled without filling your logs on every clean deploy. `shutdown_debug on_force: true` now dumps thread backtraces only when shutdown actually times out and becomes forced ([#3671](https://github.com/puma/puma/pull/3671)).

```ruby
shutdown_debug on_force: true
force_shutdown_after 30
```

That keeps the feature focused on the cases you actually want to inspect: stuck shutdowns.

Signal-based backtraces also work on more platforms now. On systems without `SIGINFO`, Puma uses `SIGPWR`, which means `pumactl info` works on Linux/JRuby too ([#3829](https://github.com/puma/puma/pull/3829)):

```sh
pumactl -S tmp/pids/puma.state info
```

And if you use `fork_worker`, Puma 8 fixes an important phased-restart correctness issue by making sure worker `0` is refreshed before later workers fork from it ([#3853](https://github.com/puma/puma/pull/3853)). That avoids descendants inheriting stale code after worker `0` has already been replaced.

### Networking defaults and JRuby performance

Puma now prefers `::` as its default wildcard bind when the host has a non-loopback IPv6 interface ([#3847](https://github.com/puma/puma/pull/3847)). The same logic is used for `port`, the Rack handler default host, and the implicit default bind. On dual-stack hosts, that gives Puma a better out-of-the-box listener for modern networks. If you need to stay explicitly IPv4-only, see the Upgrade section below.

JRuby users also get a faster HTTP parser in Puma 8 thanks to improvements that reduce byte copying, pointer chasing, and header lookup overhead ([#3838](https://github.com/puma/puma/pull/3838)). There isn't anything new to configure here; just upgrade and benefit.

## Upgrade

Check the following list to see if you're depending on any of these behaviors:

1. `port`, Rack handler defaults, and Puma's implicit default bind now prefer IPv6 `::` over `0.0.0.0` on hosts with a non-loopback IPv6 interface. If you need an IPv4 wildcard listener, set it explicitly with `port 9292, '0.0.0.0'`, `bind 'tcp://0.0.0.0:9292'`, or `Host: '0.0.0.0'` when using the Rack handler. Review firewalls, health checks, reverse proxy upstreams, and any socket activation setup that assumed `0.0.0.0`.
2. Unspecified IPv6 wildcard binds such as `bind 'tcp://[::]:9292'` and `bind 'ssl://[::]:9292?...'` are now rewritten to IPv4 `0.0.0.0` with a warning when Puma does not detect a non-loopback IPv6 interface. If you require a true IPv6-only listener, make sure the host has non-loopback IPv6 before Puma boots, or bind to a specific IPv6 address instead of `::`; if you previously relied on Puma failing fast, add your own startup check for IPv6 availability.
3. `before_thread_start` hook blocks now receive one `Puma::ServerPluginControl` argument. Update strict-arity lambdas, proc-to-method adapters, and any other zero-argument hook code to accept or ignore that parameter, for example `before_thread_start { |_control| ... }`. If you want to use the new object, it exposes helpers like `update_thread_pool_min_max`.
4. `supported_http_methods` rejections now happen during request parsing, before Puma finishes building the request environment or reading the request body. If you depended on unsupported methods reaching your app, triggering `lowlevel_error_handler`, returning a particular `501` body, sending `Expect: 100-continue`, or consuming the request body before rejection, update those expectations; if you need the old application-level behavior, allow the method in `supported_http_methods` and reject it in your app instead.
5. Requests rejected by `http_content_length_limit` now return `413` with `Connection: close`, and Puma closes the HTTP/1.1 connection after that response. Update clients, proxies, and load balancers to open a new connection after a `413` instead of trying to reuse the same socket.
6. On platforms without `SIGINFO`, Puma now traps `SIGPWR` as `Puma.backtrace_signal`, and `pumactl info` sends `SIGPWR`. If your supervisor, init system, container tooling, or operating procedures already use `SIGPWR` for something else, reconfigure that integration or stop using `pumactl info` / signal-based backtraces on those hosts; also update runbooks that assumed no backtrace signal was available there.
7. In `fork_worker` mode, phased restart and refork flows now restart worker `0` first. If your automation, logs, or hooks assumed worker `0` restarted later during `SIGUSR1`, `pumactl phased-restart`, `SIGURG`, or `pumactl refork`, update them to treat worker `0` as the first worker in the new cycle and re-check any `before_refork`, `after_refork`, `before_worker_boot`, `before_worker_shutdown`, and related lifecycle assumptions.

Then, update your Gemfile:

`gem 'puma', '< 9'`
